## 什么是 RAG？

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将**信息检索**与**大语言模型生成**相结合的技术架构。

核心思路非常简单：**在 LLM 生成回答之前，先从外部知识库中检索相关内容，将检索结果作为上下文注入 prompt，再让模型基于这些真实信息生成回答。**

### 没有 RAG 时的问题

假设你问一个 LLM："我们公司 2026 年 Q1 的营收是多少？"

模型会怎么做？它会**编造一个看起来合理的数字**。因为它根本不知道你们公司的财务数据——这些信息不在它的训练数据中。

这就是 LLM 的三个根本性局限：

1. **知识截止（Knowledge Cutoff）**：训练数据有截止日期，不知道训练完成之后发生的事
2. **幻觉（Hallucination）**：对不知道的信息会自信地编造，而非承认不知道
3. **领域缺失（Domain Gap）**：不了解私有数据——你的代码库、内部文档、客户信息、业务规则

### RAG 如何解决

同样的问题，用 RAG 的流程：

```
用户提问: "2026 Q1 营收是多少？"
    ↓
[1] 问题向量化 → 在知识库中检索
    ↓
[2] 找到 Q1 财报文档： "...2026年Q1总营收为 3.2 亿元..."
    ↓
[3] 将检索结果注入 prompt
    ↓
[4] LLM 基于真实数据回答: "根据财报数据，2026年Q1总营收为 3.2 亿元。"
```

模型不再"凭记忆"回答，而是"开卷考试"——基于真实文档生成答案。

## RAG 的工作流程

一个完整的 RAG 系统分为**离线索引**和**在线查询**两个阶段：

### 离线索引阶段（数据准备）

```
原始文档（PDF/Word/网页/代码/数据库）
    ↓ 文档加载（Document Loading）
    ↓ 文本分块（Chunking）—— 将长文档切成小块
    ↓ 向量化（Embedding）—— 将每个文本块转换为数值向量
    ↓ 存储（Indexing）—— 存入向量数据库
    ↓
向量索引（可被快速检索的知识库）
```

这一步是预先完成的，不需要每次查询都跑。当文档更新时，只需增量更新索引。

### 在线查询阶段（实时问答）

```
用户提问
    ↓ 查询向量化（将问题转换为向量）
    ↓ 向量检索（在知识库中找到最相似的文档块）
    ↓ 上下文构建（将检索结果组装进 prompt）
    ↓ LLM 生成（模型基于上下文生成回答）
    ↓
答案输出（带来源引用）
```

整个过程通常在 1-3 秒内完成。

## RAG vs 微调（Fine-tuning）

很多人会把 RAG 和微调搞混。它们解决的是不同的问题：

| 维度 | RAG | 微调（Fine-tuning） |
|------|-----|-------------------|
| **解决的问题** | 知识缺失 | 能力/风格调整 |
| **知识更新** | 实时更新索引即可 | 需要重新训练 |
| **成本** | 检索 + 推理成本 | 训练成本高 |
| **可解释性** | 高（可追溯来源） | 低（知识编码在权重中） |
| **适合场景** | 事实性问答、文档查询 | 特定风格写作、特定领域推理 |
| **数据量** | 可以管理海量文档 | 训练数据通常有限 |

**一句话总结：** RAG 是给模型"开卷考试"，微调是给模型"上新课"。它们不是替代关系，而是互补——最好的系统往往两者结合。

## RAG 的应用场景

### 企业级场景
- **内部知识库问答**：员工可以用自然语言查询公司内部文档、规章制度、技术规范
- **客户支持**：基于产品文档和 FAQ 自动回答客户问题，减少人工客服压力
- **法律/合规**：快速检索相关法条、判例、合规要求
- **医疗辅助**：基于医学文献辅助医生决策（需要严格的验证流程）

### 开发者工具
- **代码助手**：基于项目代码库回答技术问题（Cursor、GitHub Copilot 就是这个思路）
- **API 文档问答**：不用翻文档，直接问"这个接口怎么用"
- **日志分析**：从海量日志中检索关键错误信息并解释

### 内容创作
- **研究助手**：基于论文库辅助文献综述
- **写作辅助**：基于素材库生成有依据的内容

## RAG 的演进路线

RAG 技术从 2020 年提出到现在，已经经历了几个阶段：

### Naive RAG（基础 RAG）
最简单的实现：固定分块 + 向量检索 + 直接注入。适合原型验证，但检索质量有限。

### Advanced RAG（进阶 RAG）
加入了查询改写、混合检索、重排序（Re-ranking）、多跳检索等技术。显著提升检索质量。

### Modular RAG（模块化 RAG）
将 RAG 拆分为可插拔的模块——检索器、重排序器、生成器、评估器——每个模块独立优化和替换。这是当前主流架构。

### Agentic RAG（智能体 RAG）
让 Agent 自主决定何时检索、检索什么、检索几次。不再是一次性的"检索→生成"流水线，而是一个可以动态调整策略的智能系统。

## 一个最小 RAG 原型

用最少的代码实现一个能跑的 RAG：

```typescript
import { OpenAI } from "openai";
import { OpenAIEmbeddings } from "@langchain/openai";
import { MemoryVectorStore } from "langchain/vectorstores/memory";
import { Document } from "@langchain/core/documents";

// 1. 准备文档
const docs = [
  new Document({ pageContent: "2026年Q1总营收为3.2亿元，同比增长18%。" }),
  new Document({ pageContent: "2026年Q1毛利率为45.2%，较上年同期提升2个百分点。" }),
  new Document({ pageContent: "公司计划在2026年Q2推出三款新产品。" }),
];

// 2. 向量化并存入内存向量库
const vectorStore = await MemoryVectorStore.fromDocuments(
  docs,
  new OpenAIEmbeddings()
);

// 3. 检索
const query = "Q1业绩怎么样？";
const results = await vectorStore.similaritySearch(query, 2);

// 4. 构建 prompt 并调用 LLM
const context = results.map((r) => r.pageContent).join("\n");

const openai = new OpenAI();
const response = await openai.chat.completions.create({
  model: "gpt-4o",
  messages: [
    {
      role: "system",
      content: `基于以下参考信息回答问题。如果参考信息中没有答案，就说"无法从提供的信息中找到答案"。\n\n参考信息：\n${context}`,
    },
    { role: "user", content: query },
  ],
});

console.log(response.choices[0].message.content);
// 输出: "根据数据，2026年Q1总营收为3.2亿元，同比增长18%。毛利率为45.2%，较上年同期提升2个百分点。"
```

这就是 RAG 的全部核心逻辑——**检索 + 注入 + 生成**。后面的所有优化，都是在这个骨架上做增强。
